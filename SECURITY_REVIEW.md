# CropIntel Privacy And Security Review

Review date: June 26, 2026

## Scope

This review covers the code present in this repository: a static Next.js public
website exported to Firebase Hosting. The repository does not contain API
routes, server actions, Firebase client SDK usage, Firestore rules, database
queries, authentication flows, scanner implementation, model endpoints, image
upload handlers, analytics SDKs, or signup forms.

The "Try the scanner" links point to `https://cropintel-us.vercel.app`, which is
outside this repository. Privacy, authorization, uploads, rate limiting, and
model endpoint protections for that product must be verified in the repository
or deployment that serves it.

## Data Inventory Verified In This Repository

Collected directly by this static site:

- No account data is collected by the local code.
- No email address is collected by the local code.
- No farm details are collected by the local code.
- No location information is requested by the local code.
- No crop photos are uploaded or stored by the local code.
- No diagnosis history is created or stored by the local code.
- No analytics SDK is present in the local code.
- No API usage is generated by local API routes because none exist.

Sent to third parties by this static site:

- Fontshare receives browser requests for the Satoshi font stylesheet.
- Users who click scanner links are sent to the separate CropIntel scanner app
  at `https://cropintel-us.vercel.app`.
- Firebase Hosting serves the exported static files in production.

Potential data handled by the linked scanner/backend, based on product copy and
the required privacy prompt:

- Accounts, emails, authentication events, password reset activity, farm
  details, location details, uploaded crop photos, diagnosis results, diagnosis
  history, analytics, logs, and API/model usage.

## Issues Found And Fixes

| Severity | Issue | Fix | Remaining Risk |
| --- | --- | --- | --- |
| High | A real-looking `RESEND_API_KEY` was present in local `.env.local`. The file is ignored by git, but local secrets should not be used as examples or left in a shared workspace. | Replaced `.env.local` values with placeholders and added `.env.example`. Rotate the exposed Resend key in Resend before using it again. | Historical exposure cannot be ruled out from this workspace alone. Rotation is required outside the repo. |
| Medium | No public privacy policy page was available from the footer. | Added `/privacy` with a plain-language U.S. privacy policy and linked it from the footer. | The linked scanner app must also link to the policy in signup/account flows. This repo has no signup flow to update. |
| Medium | Firebase Hosting did not set common browser security headers beyond cache controls. | Added CSP, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, and `X-Frame-Options` headers in `firebase.json`. | Static Next requires `'unsafe-inline'` for inline scripts/styles in this export. A stricter nonce/hash CSP would require a server-rendered header strategy not present in this static hosting setup. |
| Low | Environment and security setup docs did not state that the website has no runtime secrets and that scanner/backend protections must live server-side. | Updated README with concrete env, rate-limit, and scanner/backend security requirements. | Actual backend controls must be implemented and tested in the scanner/backend codebase. |
| Moderate | `npm audit --audit-level=moderate` reports a PostCSS advisory through `next@16.2.9`. The suggested `npm audit fix --force` would downgrade Next to `9.3.3`, which is a breaking and insecure remediation path for this app. | Did not apply the force downgrade. Keep Next updated and apply the first compatible patched Next release when available. | Vulnerability remains until Next publishes or the project adopts a compatible patched dependency path. |

## OWASP Review Notes

- Injection: No SQL, NoSQL, database, or query-building code exists in this repo.
- XSS: No `dangerouslySetInnerHTML` or direct `innerHTML` rendering exists in
  `src`. User-controlled rendering paths were not found.
- Authentication and authorization: No auth code exists in this repo.
- Ownership checks: No farms, diagnoses, users, or database records are handled
  in this repo.
- Firestore rules: No Firestore rules or Firebase client database access exist
  in this repo.
- Exposed secrets: No committed secret files were found in tracked source. A
  local ignored `.env.local` secret was scrubbed and must be rotated.
- Server-side validation: No server-side routes exist in this repo.
- Upload safety: No upload handlers exist in this repo.
- Error details: No API or server error responses exist in this repo.
- Security headers: Added Firebase Hosting headers.
- CSRF: No state-changing authenticated routes exist in this repo.
- Redirects: No dynamic redirects exist. External scanner links use fixed
  allowlisted URLs.
- Dependency vulnerabilities: Run `npm audit` during release checks and fix
  reported vulnerable packages after validating the build.

## Required Controls For Scanner Or Backend Repositories

These controls cannot be completed in this static website because the relevant
code is not present here:

- Keep private API keys, model tokens, Firebase Admin credentials, and service
  account JSON in server-side secret stores only.
- Put costly AI/model calls behind authenticated backend endpoints.
- Validate every request body with an allowlist schema.
- Enforce ownership checks for accounts, farms, diagnoses, photos, and history.
- Validate uploads by size, MIME type, file signature, dimensions, and allowed
  extensions before storage or model processing.
- Store uploaded files in private buckets unless a specific object is intended
  to be public.
- Add rate limits by authenticated user ID and IP address for diagnosis, image
  uploads, login, signup, password reset, and model routes.
- Return `429` responses with clear user-facing text and no internal rate-limit
  implementation details.
- Avoid returning stack traces, provider errors, tokens, prompts, bucket paths,
  signed URLs, or model internals to browsers.
- Use CSRF protection for cookie-authenticated state-changing requests.
- Review Firestore or database rules so users can read, update, and delete only
  their own data unless explicitly shared.
